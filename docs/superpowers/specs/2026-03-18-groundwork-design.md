# Groundwork — Design Spec

A developer tool that lets vibecoders define their database schema once in plain English, then generates a persistent AI context file (GROUNDWORK.md) that produces consistent, reliable database queries and structures in any AI coding session.

## Decisions

| Decision | Choice |
|----------|--------|
| Parser scope | Basic tables, columns with types, foreign key relationships |
| API key | Environment variable (`ANTHROPIC_API_KEY` in `.env.local`) |
| Storage | In-memory only (lost on restart) |
| Page flow | Single-page stepper (Describe → Review → Export) |
| Layout | Left sidebar with step progress indicator |
| Visual style | Soft dark (#191919) with Notion typography (Inter/system) |

## Stack

- Next.js 14 (App Router)
- Tailwind CSS
- Anthropic SDK (`@anthropic-ai/sdk`)
- Zod (response validation)
- TypeScript

## Architecture

### Data Flow

```
Plain English → Claude API (parse) → Zod validation → Schema Object → Generator → GROUNDWORK.md
```

### API Routes

- `POST /api/parse-schema` — takes `{ input: string }`, calls Claude, validates response with Zod, returns `{ schema: Schema }`
- `POST /api/generate-context` — takes `{ schema: Schema }`, returns `{ content: string }` (the GROUNDWORK.md markdown)

Note: `GET /api/schemas` was removed — the prototype has no schema history feature, and in-memory storage makes it low-value. The single-page flow passes the schema object directly between steps via React state.

## Schema Object

```typescript
type ColumnType = "string" | "integer" | "timestamp" | "boolean" | "text" | "float";

interface Schema {
  id: string;                  // generated UUID
  name: string;                // inferred from input (e.g. "SaaS App")
  description: string;         // brief summary
  tables: Table[];
  relationships: Relationship[];
  createdAt: string;           // ISO timestamp
}

interface Table {
  name: string;                // e.g. "users"
  description: string;         // what this table represents
  columns: Column[];
}

interface Column {
  name: string;                // e.g. "email"
  type: ColumnType;
  primaryKey?: boolean;
  nullable?: boolean;
  unique?: boolean;
  defaultValue?: string;       // e.g. "now()", "true"
  references?: {               // foreign key reference
    table: string;
    column: string;
  };
}

interface Relationship {
  from: string;                // table name, e.g. "posts"
  to: string;                  // table name, e.g. "users"
  type: "one-to-many" | "many-to-many" | "one-to-one";
  foreignKey: string;          // e.g. "user_id"
  description: string;         // e.g. "Each post belongs to one user"
}
```

### Zod Validation

Claude's JSON response is validated with a Zod schema matching the interfaces above. If Claude returns a column type outside the `ColumnType` union (e.g. `"varchar"`), it defaults to `"string"`. This ensures the schema object always conforms to the expected shape regardless of Claude's output.

## GROUNDWORK.md Output Format

The generator produces concrete SQL — not descriptions. This is what makes AI actually produce correct queries when given the context file.

```markdown
# GROUNDWORK.md
# Generated by Groundwork

## Schema: [Schema Name]
[Description]

---

## Tables

### [table_name]
[Table description]

| Column     | Type      | Constraints              |
|------------|-----------|--------------------------|
| id         | integer   | primary key              |
| name       | string    |                          |
| email      | string    | unique                   |
| user_id    | integer   | fk → users.id            |
| created_at | timestamp | default: now()           |

(repeated for each table)

---

## Relationships
- **[from] → [to]**: [type] via `[from].[foreignKey]` — [description]

---

## AI Instructions

When working with this database schema, follow these rules:

1. Always use the exact table and column names defined above
2. Always include proper foreign key references in queries
3. Use the relationships defined above — do not infer new ones
4. When creating new queries, follow these patterns:

### Example Queries

**[table_name]**

SELECT * FROM [table_name];

SELECT * FROM [table_name] WHERE id = ?;

INSERT INTO [table_name] ([non-pk columns]) VALUES ([?...]);

**[table with relationships]**

SELECT t.*, r.name as [related]_name
FROM [table] t
JOIN [related] r ON t.[fk] = r.id
WHERE t.id = ?;

(concrete SQL generated per table from column names and relationships)

---

## Common Patterns

- Fetching with relations: JOIN using defined foreign keys
- Inserting records: Include all non-nullable columns; let defaults handle timestamps
- Filtering: Use indexed columns (primary keys, foreign keys) for WHERE clauses

---

## Rules
- This file is the source of truth for database structure
- Do not modify table names, column names, or types without updating this file
- All queries should be consistent with the schema defined here
- When in doubt, reference this file before writing any database code
```

Key design decisions:
- Human-readable markdown tables, not JSON/YAML
- Concrete SQL queries per table (not descriptions) — this is what makes AI produce correct output
- Foreign key columns show `fk → table.column` in the constraints column
- Rules section tells AI models to treat this as authoritative
- No framework-specific code — works with Prisma, Drizzle, raw SQL, Django, etc.
- Generator uses string templates, no AI call — fast and deterministic

## Claude API Integration

One AI call in the entire app: schema parsing.

**Model:** `claude-sonnet-4-20250514`
**Max tokens:** 2000

### Request Format

```typescript
const response = await anthropic.messages.create({
  model: "claude-sonnet-4-20250514",
  max_tokens: 2000,
  system: SYSTEM_PROMPT,       // see below
  messages: [
    {
      role: "user",
      content: input            // raw user input from textarea, sent as-is
    }
  ]
});

// Extract text from Claude's response
const json = response.content[0].text;
```

### System Prompt

```
You are a database schema parser. Given a plain English description
of a database, extract the structured schema as JSON.

Return ONLY valid JSON matching this exact shape:
{
  "name": string,
  "description": string,
  "tables": [
    {
      "name": string,
      "description": string,
      "columns": [
        {
          "name": string,
          "type": "string" | "integer" | "timestamp" | "boolean" | "text" | "float",
          "primaryKey": boolean (optional),
          "nullable": boolean (optional),
          "unique": boolean (optional),
          "defaultValue": string (optional),
          "references": { "table": string, "column": string } (optional)
        }
      ]
    }
  ],
  "relationships": [
    {
      "from": string,
      "to": string,
      "type": "one-to-many" | "many-to-many" | "one-to-one",
      "foreignKey": string,
      "description": string
    }
  ]
}

Rules:
- Infer reasonable column types from the six allowed values above
- Add an "id" integer primary key to every table if not mentioned
- Add "created_at" timestamp with default "now()" to every table if not mentioned
- Infer foreign keys from relationships (e.g. "posts belong to users" → posts.user_id with references to users.id)
- If the description is vague, make reasonable assumptions and include them
- Table and column names should be snake_case
- Do not include any text outside the JSON object
```

### Error Handling

| Error | Handling |
|-------|----------|
| Empty/short input (< 10 chars) | 400 — client-side validation, no API call |
| Input too long (> 5000 chars) | 400 — client-side validation with message |
| Missing `ANTHROPIC_API_KEY` | 500 — `"Set ANTHROPIC_API_KEY in .env.local"` |
| Invalid JSON from Claude | Retry once with same prompt, then 500 — `"Couldn't parse that — try being more specific"` |
| Zod validation failure | 500 — `"Schema parsing produced unexpected structure — try rephrasing"` |
| Rate limit (429) / network / other API errors | 500 — `"Something went wrong — please try again"` |

`page.tsx` manages these state variables, passed as props to step components:

```typescript

```

State is lost on browser refresh — acceptable for a prototype.

### Step 1 — Describe
- Large auto-expanding textarea with placeholder example (JS resize handler on input)
- "Parse Schema →" button
- Loading spinner on button while parsing, textarea disabled
- Inline error message on failure

### Step 2 — Review
- Schema name and description at top
- Responsive card grid (1 col mobile, 2 col desktop) — one card per table
- Each card: table name header, columns with type badges, relationship indicators
- "← Edit" link to return to Step 1 with input pre-filled
- "Generate Context →" button with loading spinner while generating
- Inline error on generation failure

### Step 3 — Export
- GROUNDWORK.md in a styled code block (monospace)
- Copy to Clipboard + Download .md buttons side by side
- "← Start Over" link to reset all state back to Step 1

### Components
- `page.tsx` — all state, renders current step

- `SchemaReview.tsx` — cards grid (Step 2)
